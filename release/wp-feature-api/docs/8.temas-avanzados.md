## Sistema de Gestión de Versiones

La API de Funcionalidades de WordPress incluye un sistema de gestión de versiones integrado que permite que múltiples plugins incluyan su propia copia de la API sin conflictos.

### Cómo Funciona

| Componente | Descripción |
|------------|-------------|
| **Registro de Versiones** | Cuando un plugin incluye la API de Funcionalidades, la versión se registra automáticamente en un registro global. |
| **Resolución de Versiones** | Durante el hook `plugins_loaded`, el sistema determina la versión más alta disponible y solo inicializa completamente esa versión. |
| **Capa de Compatibilidad** | Todos los plugins registran sus funcionalidades con la versión activa (la más alta), independientemente de qué versión hayan incluido. |

### Proceso de Inicialización

```php
// 1. Cada plugin incluye su copia de la API de Funcionalidades
require_once __DIR__ . '/vendor/automattic/wp-feature-api/wp-feature-api.php';

// 2. Cuando se carga, cada versión se registra usando wp_feature_api_register_version()
// 3. El resolutor de versiones (wp_feature_api_version_resolver()) identifica la versión más alta
// 4. El sistema priorizará una instalación directa de la API de Funcionalidades sobre una version empaquetada
//    si ambas tienen la misma cadena de versión
// 5. Solo el código de esa versión se inicializa completamente
// 6. El action wp_feature_api_init se dispara cuando la inicialización está completa
// 7. Todos los plugins registran sus funcionalidades usando este hook
add_action('wp_feature_api_init', 'mi_plugin_registrar_funcionalidades');
```

## Repositorios (`WP_Feature_Repository_Interface`)

La API de Funcionalidades usa un patrón de repositorio para almacenar y recuperar funcionalidades.

### Interfaz del Repositorio

```php
interface WP_Feature_Repository_Interface {
    public function save( WP_Feature $feature ): bool;
    public function delete( string $id ): bool;
    public function find( string $id ): ?WP_Feature;
    public function query( WP_Feature_Query $query ): array;
    public function get_all(): array;
    // ... otros métodos
}
```

### Repositorios Disponibles

| Repositorio | Descripción |
|-------------|-------------|
| `WP_Feature_Repository_Memory` | Almacenamiento en memoria simple, no persistente, activo durante una sola solicitud PHP. |
| **Personalizado** | Los desarrolladores pueden crear clases de repositorio personalizadas implementando la interfaz para almacenar funcionalidades de manera diferente. |

### Ejemplo: Repositorio Personalizado

```php
<?php
class WP_Feature_Repository_Custom implements WP_Feature_Repository_Interface {
    // Implementar métodos para almacenar en base de datos, CPT, etc.
    public function save( WP_Feature $feature ): bool {
        // Guardar en base de datos personalizada
    }
    
    public function find( string $id ): ?WP_Feature {
        // Recuperar de la base de datos
    }
    
    public function query( WP_Feature_Query $query ): array {
        // Consultar con SQL eficiente
    }
}

// Registrar el repositorio personalizado
add_filter( 'wp_feature_repository', function( $default_repository ) {
    return new WP_Feature_Repository_Custom();
} );
```

### Consultas Nativas vs PHP

El método `query()` maneja el filtrado. El repositorio de memoria por defecto realiza filtrado básico en PHP. Repositorios más avanzados (como uno respaldado por base de datos) podrían implementar consultas más eficientes (cláusulas SQL `WHERE`, búsqueda de similitud vectorial si se añaden embeddings).

El método `supports_native_query()` permite a un repositorio indicar si puede manejar ciertos tipos de consulta más eficientemente que el filtrado PHP por defecto.

## Adaptadores de Esquema (`WP_Feature_Schema_Adapter`)

Diferentes consumidores, especialmente LLMs como los modelos de OpenAI, tienen requisitos específicos o limitaciones respecto al formato JSON Schema para llamadas de función/herramienta.

### Propósito

La clase `WP_Feature_Schema_Adapter` proporciona un mecanismo para transformar el JSON Schema estándar definido para una funcionalidad en un formato compatible con un consumidor específico.

### Adaptador por Defecto (OpenAI)

La clase base incluye reglas basadas en los requisitos documentados de OpenAI:

- Eliminar palabras clave no soportadas
- Hacer todas las propiedades requeridas o anulables
- Establecer `additionalProperties: false`

### Crear un Adaptador Personalizado

```php
<?php
class Mi_Adaptador_Schema extends WP_Feature_Schema_Adapter {
    protected function transform_input_schema( array $schema ): array {
        // Transformaciones personalizadas para el consumidor
        $schema['additionalProperties'] = false;
        return $schema;
    }
    
    protected function transform_output_schema( array $schema ): array {
        // Transformaciones de salida personalizadas
        return $schema;
    }
}

// Registrar el adaptador
add_filter( 'wp_feature_input_schema_adapter', function() {
    return 'Mi_Adaptador_Schema';
} );

add_filter( 'wp_feature_output_schema_adapter', function() {
    return 'Mi_Adaptador_Schema';
} );
```

### Filtros Disponibles

| Filtro | Descripción |
|--------|-------------|
| `wp_feature_input_schema_adapter` | Especifica la clase de adaptador para esquemas de entrada. |
| `wp_feature_output_schema_adapter` | Especifica la clase de adaptador para esquemas de salida. |

## Composabilidad (Conceptual)

La idea de que las funcionalidades pueden llamar a otras funcionalidades.

### Concepto

Similar a cómo las funciones llaman a otras funciones, una funcionalidad compleja podría construirse orquestando llamadas a varias funcionalidades más simples y registradas.

### Implementación

Esto no está explícitamente construido en la estructura actual de la API pero podría implementarse dentro del `callback` de una funcionalidad. El callback podría usar `wp_find_feature()` y `$feature->call()` para ejecutar otras funcionalidades según sea necesario.

### Ejemplo de Composabilidad

```php
<?php
function mi_funcionalidad_compuesta_callback( $context ) {
    // Paso 1: Obtener información del sitio
    $info_sitio = wp_find_feature( 'recurso-demo/info-sitio' );
    $datos_sitio = $info_sitio->call();
    
    // Paso 2: Crear un análisis basado en los datos
    $analisis = array(
        'version' => $datos_sitio['version'] ?? 'desconocida',
        'plugins_activos' => count( $datos_sitio['plugins_activos'] ?? [] ),
    );
    
    // Paso 3: Registrar el análisis como una nueva funcionalidad (dinámicamente)
    wp_register_feature( array(
        'id' => 'mi-analisis/' . uniqid(),
        'name' => __( 'Análisis Generado', 'mi-plugin' ),
        'description' => __( 'Análisis generado automáticamente.', 'mi-plugin' ),
        'type' => WP_Feature::TYPE_RESOURCE,
        'callback' => function() use ( $analisis ) {
            return $analisis;
        },
        'permission_callback' => '__return_true',
    ) );
    
    return $analisis;
}
```

### Beneficios

| Beneficio | Descripción |
|-----------|-------------|
| **Modularidad** | Construir funcionalidades complejas a partir de bloques más pequeños. |
| **Reutilización** | Reutilizar funcionalidades granulares definidas una vez. |
| **Mantenibilidad** | Facilitar el mantenimiento y las pruebas de funcionalidades individuales. |
