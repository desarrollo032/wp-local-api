# 12. Proceso de Lanzamiento de la API de Funcionalidades de WordPress

> **Documento que describe el proceso de lanzamiento para la API de Funcionalidades de WordPress, que maneja múltiples formatos de distribución para llegar a diferentes tipos de usuarios.**

---

## Formatos de Distribución

La API de Funcionalidades de WordPress se distribuye en tres formatos:

| Formato | Audiencia | Descripción |
|---------|-----------|-------------|
| **ZIP del Plugin de WordPress** | Propietarios de sitios WordPress | Para usuarios que desean instalar el plugin directamente en su sitio WordPress. |
| **Paquete Composer** | Desarrolladores de plugins de WordPress | Para desarrolladores que desean incluir la API en sus plugins como dependencia. |
| **Paquete NPM** | Desarrolladores JavaScript | Para desarrolladores que desean usar el SDK del lado del cliente en sus aplicaciones. |

---

## Proceso de Lanzamiento Automatizado

Usamos **GitHub Actions** para automatizar el proceso de lanzamiento. El workflow se activa cuando se publica un nuevo release en GitHub.

### Cómo Crear un Release

#### Paso 1: Preparar los Cambios

Asegúrate de que todos los cambios estén fusionados en la rama principal (`trunk`).

#### Paso 2: Actualizar Números de Versión

Actualiza los números de versión en los siguientes archivos:

| Archivo | Qué Actualizar |
|---------|----------------|
| `wp-feature-api.php` | `$wp_feature_api_version` y cabecera del plugin |
| `package.json` | `version` |
| `packages/*/package.json` | `version` en cada subpaquete |

#### Paso 3: Push de Cambios

Empuja los cambios de versión a la rama principal.

```bash
git add wp-feature-api.php package.json packages/*/package.json
git commit -m "Bump version to X.X.X"
git push origin trunk
```

#### Paso 4: Crear el Release en GitHub

1. Ve al repositorio de GitHub.
2. Haz clic en **Releases** > **Draft a new release**.
3. Elige un tag que coincida con la versión (ej: `v0.1.3`).
4. Añade un título y descripción detallando los cambios.
5. Publica el release.

---

## ¿Qué Ocurre Después de Publicar?

Cuando publicas un release, el workflow de GitHub Actions ejecutará las siguientes acciones:

### 1. Construir el Proyecto

| Acción | Descripción |
|--------|-------------|
| **Instalar Dependencias** | Instala todas las dependencias de npm y composer. |
| **Construir Assets JavaScript** | Compila y minifica los bundles JavaScript y assets. |

```bash
# Ejemplo de comandos ejecutados
npm install
composer install
npm run build
```

### 2. Crear ZIP del Plugin de WordPress

| Acción | Descripción |
|--------|-------------|
| **Empaquetar Plugin** | Agrupa todos los archivos necesarios en un ZIP del plugin de WordPress. |
| **Subir como Asset** | Sube el ZIP como asset del release. |
| **Crear ZIP de Demo** | También crea un ZIP para el plugin de demostración. |

### 3. Actualizar Tag para Composer

| Acción | Descripción |
|--------|-------------|
| **Añadir Archivos Construidos** | Añade los archivos de construcción al tag del release. |
| **Packagist** | Packagist usará este tag con los archivos de construcción incluidos. |

### 4. Publicar Paquetes NPM

| Acción | Descripción |
|--------|-------------|
| **Publicar Paquetes Client** | Publica los paquetes del cliente en npm. |

## Flujo de Trabajo de GitHub Actions

El archivo de workflow (`.github/workflows/release.yml`) define los siguientes trabajos:

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '20'
          registry-url: 'https://registry.npmjs.org'
      
      - name: Setup PHP
        uses: shivammathur/setup-php@v2
        with:
          php-version: '8.0'
      
      - name: Install Dependencies
        run: |
          composer install --no-dev
          npm ci
      
      - name: Build
        run: npm run build
      
      - name: Create Plugin ZIP
        run: npm run create-zip
      
      - name: Upload Release Asset
        uses: actions/upload-release-asset@v1
        with:
          upload_url: ${{ steps.create_release.outputs.upload_url }}
          asset_path: build/wp-feature-api.zip
          asset_name: wp-feature-api.zip
          asset_content_type: application/zip
      
      - name: Publish to NPM
        run: npm publish
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
```

## Verificación del Release

Después de publicar un release, verifica que:

| Verificación | Cómo |
|--------------|------|
| **ZIP del Plugin** | Descarga y verifica el ZIP desde la página de releases. |
| **Paquete Composer** | Verifica en [Packagist](https://packagist.org/packages/automattic/wp-feature-api). |
| **Paquetes NPM** | Verifica en [npmjs.com](https://www.npmjs.com/~automattic). |
| **Tags Git** | Verifica que el tag existe: `git tag -l "v*"` |

---

## Notas Importantes

> ⚠️ **Versiones Semánticas:** Seguimos el versionado semántico (SemVer). Los cambios breaking incrementan la versión mayor, las nuevas funcionalidades la menor, y los bugs la patch.

> 📦 **Archivos Construidos:** Los archivos construidos (minificados, transpilados) se incluyen en el repositorio para facilitar la distribución.

> 🔄 **Tags Inmutables:** Una vez creado un tag, no lo modifiques. Si necesitas corregir algo, crea un nuevo release.
