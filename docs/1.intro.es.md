### 1. Introducción y visión general

**¿Qué es la Feature API?**

La WordPress Feature API ofrece una forma estandarizada de registrar y descubrir unidades de funcionalidad en un sitio WordPress. Estas unidades, llamadas "Features", pueden representar acciones de obtención de datos (**Resources**) o acciones de modificación/creación de datos (**Tools**).

Actúa como un registro central, facilitando que distintas partes de WordPress, plugins de terceros, temas y sistemas externos (como agentes de IA) entiendan e interactúen con las capacidades disponibles en un sitio concreto.

**Conceptos clave**

* **Feature:** Una pieza concreta de funcionalidad. Tiene un ID único, nombre y descripción legibles por humanos, definiciones de entrada/salida (usando JSON Schema), permisos y el código asociado (callback o alias REST).
* **Resource:** Tipo de Feature usado para *recuperar* datos (operaciones de solo lectura). Normalmente corresponde a peticiones `GET` en la REST API.
* **Tool:** Tipo de Feature usado para *realizar acciones* o *modificar* datos (operaciones de escritura/creación/actualización/eliminación). Normalmente corresponde a `POST`, `PUT`, `PATCH`, `DELETE`.
* **Registry:** Objeto central singleton (`WP_Feature_Registry`) que contiene todas las features registradas y ofrece métodos para registrar, eliminar, buscar y consultar features.
* **Repository:** Mecanismo de almacenamiento subyacente para el registro (p. ej., en memoria o base de datos). La API define una interfaz (`WP_Feature_Repository_Interface`) para soportar distintos backends; por defecto se usa almacenamiento en memoria (`WP_Feature_Repository_Memory`).
* **Callback:** La función o método PHP que se ejecuta cuando se invoca una Feature.
* **Schema:** Definiciones JSON Schema para la entrada esperada (`input_schema`) y la salida (`output_schema`) de una Feature. Esto permite validación y ayuda a los agentes a entender cómo usarla.
* **Category:** Mecanismo para agrupar features relacionadas para facilitar organización y descubrimiento.
* **Elegibilidad (`is_eligible`):** Callback que determina si una feature debe estar disponible en el contexto actual (p. ej., requiere que otro plugin esté activo).

**Objetivos y beneficios**

* **Estandarización:** Proporciona una forma única y consistente de exponer capacidades del sitio.
* **Descubribilidad:** Permite a sistemas (agentes IA o componentes UI) localizar fácilmente acciones y fuentes de datos disponibles.
* **Contexto:** Las features pueden estar condicionadas al entorno o usuario (`is_eligible`).
* **Reutilización:** Una feature definida una vez puede ser consumida por múltiples consumidores (llamadas PHP directas, peticiones REST, agentes IA).
* **Extensibilidad:** Está diseñada para que plugins y temas registren sus propias features junto con las del core.
* **Base para IA:** Proporciona bloques de construcción (tools y resources) útiles para agentes IA que operen dentro o interactúen con WordPress.

**Relación con MCP (Model Context Protocol)**

La Feature API **no es** MCP. MCP es una especificación de protocolo para cómo agentes IA y capacidades interactúan entre sistemas.

La Feature API es un sistema nativo de WordPress para registrar y gestionar capacidades dentro de un único sitio. Está diseñada para ser **compatible** con MCP mediante un adaptador:

1. Plugins, temas y core registran funcionalidades como Features usando esta API.
2. Un **MCP Adapter** (plugin o componente externo) lee las features del registro de la Feature API y las expone como Tools/Resources mediante el protocolo MCP.
3. Así se separa la integración específica de WordPress (Feature API) de la interoperabilidad entre sistemas (MCP).
